Exasol SQL Dialect4 issues · #01 – #04
01

DATE_TRUNC argument order is reversed from PostgreSQL

Exasol puts the unit string first, then the timestamp value. All six occurrences used the wrong order.

SKILL FILEstg_iot__sensor_readings.sqlstg_iot__downtime_events.sqlstg_erp__production_orders.sqlmart_production_summary.sqlsetup_ai_tables.pyfactory_ai_agent.py
Tried — value-first (PostgreSQL style)
DATE_TRUNC(ts_column, 'HOUR') DATE_TRUNC(started_at, 'DAY')
Error
Feature not supported: date_trunc with wrong type
Fix — unit-first
DATE_TRUNC('HOUR', ts_column) DATE_TRUNC('DAY', started_at)
02

Timestamp subtraction cannot be cast to FLOAT — use SECONDS_BETWEEN()

Exasol 2026.1.0 does not allow casting timestamp interval arithmetic to a numeric type. A dedicated function is required.

stg_iot__downtime_events.sqlstg_erp__production_orders.sqlsetup_ai_tables.py
Tried
CAST(ended_at - started_at AS FLOAT) / 3600.0 AS duration_hrs
Error
Cannot cast from TIMESTAMP to DOUBLE
Fix
CAST(SECONDS_BETWEEN(started_at, ended_at) / 3600.0 AS DECIMAL(8,4)) AS duration_hrs
Note
SECONDS_BETWEEN returns an INTEGER (total seconds). Dividing by 3600.0 (a float literal) forces decimal arithmetic. Wrapping in CAST ensures the correct output precision.
03

DATEADD does not exist — use ADD_DAYS / ADD_MONTHS / ADD_YEARS

Exasol has no DATEADD function. Date arithmetic uses discrete ADD_DAYS, ADD_MONTHS, ADD_YEARS functions.

SKILL FILEmart_ai_maintenance_queue.sqlfactory_ai_agent.py
Tried (SQL Server / Snowflake style)
DATEADD('DAY', -7, shift_date)
Error
function or script DATEADD not found
Fix
ADD_DAYS(shift_date, -7)
04

Window function inside an aggregate expression is not allowed

MAX(...) OVER () nested inside AVG(CASE WHEN ...) fails. The window result must be pre-computed in a separate CTE.

mart_ai_maintenance_queue.sqlfactory_ai_agent.py
Tried — window inside aggregate
SELECT machine_id, AVG(CASE WHEN shift_date >= ADD_DAYS(MAX(shift_date) OVER (), -7) THEN oee END) AS oee_last_7d FROM MARTS.MART_OEE_DAILY GROUP BY machine_id
Error
Analytic function not allowed here
Fix — split into two CTEs
WITH oee_max_date AS ( SELECT machine_id, MAX(shift_date) AS max_date FROM MARTS.MART_OEE_DAILY GROUP BY machine_id ), oee_window AS ( SELECT o.machine_id, AVG(CASE WHEN o.shift_date >= ADD_DAYS(d.max_date, -7) THEN o.oee END) AS oee_last_7d FROM MARTS.MART_OEE_DAILY o JOIN oee_max_date d ON o.machine_id = d.machine_id GROUP BY o.machine_id )

pyexasol Python API6 issues · #05 – #10
05

Default fetchall() returns tuples — dict access requires fetch_dict=True at connect time

Unlike psycopg2's RealDictCursor, row format is not per-query in pyexasol. It must be set at connection creation.

setup_ai_tables.pyfactory_ai_agent.pysetup_exasol.py
Tried
con = pyexasol.connect(dsn=..., user=..., password=...) row = con.execute("SELECT machine_id FROM ...").fetchone() print(row["machine_id"]) # dict-style access
Error
TypeError: tuple indices must be integers or slices, not str
Fix
con = pyexasol.connect( dsn=..., user=..., password=..., websocket_sslopt={"cert_reqs": 0}, fetch_dict=True # must be set here, not per-query )
06

All column names in dict rows are UPPERCASE regardless of SQL alias casing

Even aliases written in lowercase (AS event_date) come back as EVENT_DATE. Affects every dict key access in the codebase.

setup_ai_tables.pyfactory_ai_agent.py
Tried — lowercase keys after enabling fetch_dict
row["machine_id"] # KeyError row["avg_temp_c"] # KeyError # Actual keys: ['MACHINE_ID', 'AVG_TEMP_C', 'MAX_TEMP_C', ...]
Error
KeyError: 'machine_id'
Fix — normalize to lowercase at fetch time
rows = con.execute("SELECT machine_id, avg_temp_c FROM ...").fetchall() rows = [{k.lower(): v for k, v in r.items()} for r in rows] # Now row["machine_id"] works as written
07

import_from_iterable table argument must be a (schema, table) tuple — not a dotted string

pyexasol validates identifiers strictly. A dot-separated string is rejected as unsafe.

setup_exasol.pysetup_ai_tables.py
Tried
con.import_from_iterable(data, "IOT_RAW.SENSOR_READINGS")
Error
Value [IOT_RAW.SENSOR_READINGS] is not a safe identifier
Fix
con.import_from_iterable(data, ("IOT_RAW", "SENSOR_READINGS"))
08

import_from_iterable does not support a 'columns' keyword argument in this version

The parameter appears in some versions of the pyexasol docs but is absent in the installed version. Workaround: remove IDENTITY from PK, supply IDs from Python.

setup_ai_tables.py
Tried
con.import_from_iterable( batch, ("AI_SCHEMA", "FAILURE_PATTERNS"), columns=["machine_id", "machine_type", ...] # skip IDENTITY col )
Error
ExaConnection.import_from_iterable() got an unexpected keyword argument 'columns'
Fix — remove IDENTITY, include PK from enumerate()
-- DDL: plain INT, no IDENTITY CREATE TABLE AI_SCHEMA.FAILURE_PATTERNS ( pattern_id INT PRIMARY KEY, -- not INT IDENTITY ... ) -- Python: assign IDs from enumerate for pid, row in enumerate(rows, start=1): batch.append((pid, row["machine_id"], row["machine_type"], ...)) con.import_from_iterable(batch, ("AI_SCHEMA", "FAILURE_PATTERNS"))
09

import_from_iterable hangs indefinitely when the target table has an IDENTITY column

No error is thrown — the call simply never returns. Happens when the data tuple omits the IDENTITY column. Must kill the process.

setup_exasol.py
Situation
-- Table with IDENTITY PK CREATE TABLE IOT_RAW.DOWNTIME_EVENTS ( event_id INT IDENTITY PRIMARY KEY, machine_id INT, started_at TIMESTAMP, ... ) -- Inserting rows that omit event_id
Behavior
No error. import_from_iterable() never returns. Process must be killed manually.
Fix — same as #08: remove IDENTITY, assign IDs in Python
rows_with_id = [ (i, machine_id, started_at, ended_at, reason_code) for i, (machine_id, started_at, ended_at, reason_code) in enumerate(source_rows, 1) ] con.import_from_iterable(rows_with_id, ("IOT_RAW", "DOWNTIME_EVENTS"))
10

execute() params must be a dict — ? placeholders are not supported

pyexasol uses Python's string.Formatter internally. Passing a tuple raises TypeError. The format uses {name} or {name!Q} style, not ? or %s.

factory_ai_agent.py
Tried — DBAPI-style ? with tuple
con.execute( "INSERT INTO t (a, b, c) VALUES (?, ?, ?)", (val_a, val_b, val_c) )
Error
TypeError: string.Formatter.format() argument after ** must be a mapping, not tuple
Fix — f-string with manual quote-escaping
def _sql_str(s): return "'" + str(s or "").replace("'", "''") + "'" con.execute(f""" INSERT INTO AI_SCHEMA.MAINTENANCE_RECOMMENDATIONS (machine_id, machine_name, ...) VALUES ({int(machine["machine_id"])}, {_sql_str(machine["machine_name"])}, ...) """)
Note
pyexasol's native parameterization uses {name!Q} (quoted string) and {name!i} (identifier). That was not used here due to uncertainty about the conversion behavior in this version — f-string with explicit escaping is unambiguous.

11

Exasol's internal JVM cannot resolve Docker service names

When Exasol creates a JDBC connection outward to another container, the JVM process runs in an isolated network context that does not participate in Docker's embedded DNS.

setup_exasol.py
Tried
jdbc:postgresql://mfg_postgres:5432/manufacturing_erp
Error
java.net.UnknownHostException: mfg_postgres
Fix — detect container IP at setup time via docker inspect
import subprocess r = subprocess.run( ["docker", "inspect", "mfg_postgres", "--format", "{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}"], capture_output=True, text=True ) pg_ip = r.stdout.strip() # e.g. "172.25.0.3" # Connection string: jdbc:postgresql://172.25.0.3:5432/manufacturing_erp

13

ETL-1014: JDBC driver registration requires settings.cfg in drivers/jdbc/ — not BucketFS root

Exasol's ETL framework scans a specific subpath for driver metadata. A JAR at the root is visible to scripts but not registered as a JDBC driver for Virtual Schema use.

con.execute(""" CREATE OR REPLACE JAVA ADAPTER SCRIPT ADAPTER_SCRIPTS.JDBC_ADAPTER AS %scriptclass com.exasol.adapter.RequestDispatcher; %jar /buckets/bfsdefault/default/adapter.jar; %jar /buckets/bfsdefault/default/drivers/jdbc/postgresql.jar; /""")

JDBC JAR at BucketFS root does not satisfy driver registration — must be in drivers/jdbc/

The ETL framework only scans the drivers/jdbc/ subdirectory. Root-level JARs are accessible in script %jar paths but are invisible to the JDBC driver registry.

setup_exasol.py
Tried — root upload

Result
ETL-1014 on Virtual Schema query (driver not found despite JAR being present)
Fix
curl -k -X PUT https://w:PASS@localhost:2581/default/drivers/jdbc/postgresql-42.7.3.jar -T file.jar # Script path: /buckets/bfsdefault/default/drivers/jdbc/postgresql-42.7.3.jar

dbt prefixes all schema names with target.schema unless a generate_schema_name macro is added

Default dbt behavior: a model in schema STAGING becomes DBT_MFG_STAGING. The macro overrides this to use the custom_schema_name verbatim.

macros/generate_schema_name.sql
Default behavior
-- dbt_project.yml: schema: STAGING -- Actual schema created: DBT_MFG_STAGING (target.schema prepended)
Fix — macros/generate_schema_name.sql
{% macro generate_schema_name(custom_schema_name, node) -%} {%- if custom_schema_name is none -%} {{ target.schema | trim | upper }} {%- else -%} {{ custom_schema_name | trim | upper }} {%- endif -%} {%- endmacro %}

Unicode characters in print() crash the process on Windows PowerShell with cp1252 encoding

Box-drawing characters (──), em dashes (—), and degree signs (°) written in Python string literals raise UnicodeEncodeError when stdout is bound to the Windows default cp1252 codepage.

factory_ai_agent.py

Fix — reconfigure stdout at script start
import sys # Near the top of any script that prints Unicode if hasattr(sys.stdout, "reconfigure"): sys.stdout.reconfigure(encoding="utf-8", errors="replace") # Or, before running from PowerShell: # $env:PYTHONIOENCODING = "utf-8"
